## 3 点セット

::info-card{title="3点セット" icon="mdi:format-list-checks"}
問題を評価するための 3 つの観点：

### 🧑 誰が

- ユーザー
- 開発者
- レビュアー
- 運用者
- CS 担当
- 営業

### ⏰ どんなときに

- 機能を使う
- 問い合わせる
- 実装する
- テストする
- レビューする
- デプロイする
- 障害対応する
- オンボーディングする

### 📊 どれくらい

- **時間**: 分/時間/日
- **頻度**: 毎回/週 N 回/月 N 回
- **件数**: 月 N 件
::

## 使い方

::alert{type="success"}
✍️ **Problem が出たらこの 3 点で説明してみる**

埋められないなら「気になること」として保留し、本当に困るまで待つ
::

## 「べき論」と実害のギャップに注意

::alert{type="warning"}
「コードは新しい書き方であるべき」「手作業は自動化すべき」「テストカバレッジは高くあるべき」など、一般的なベストプラクティスや理想像があります。

これらに照らすと「問題」に見えても、実際の文脈では頻度・影響・安定性を考慮すると実害がないこともあります。

Problem が出たら「一般的には良くないとされているが、この文脈で本当に困っているか？」を確認しましょう。
::

## 実例で確認

### 実は影響が大きい例 🔥

::before-after
#before
「問い合わせ対応が手作業で面倒」

#after
「**ユーザーが**、問い合わせ後に、**回答まで平均 3 日待っている**（**月 50 件**）」

→ 誰が・どんなときに・どれくらいが明確で、影響が大きいことがわかる
::

### 実は影響が小さい例 🤔

::before-after
#before
「この機能のコードが古い書き方のまま」

#after
「**年 1 回の更新時** しか触らず、動作も安定しているので実害なし → **保留**」

→ 頻度が低く、実害がないため優先度は低い
::

## まとめ

::key-points{title="3点セットで得られる効果"}
- 本当に対処すべき問題が明確になる
- 優先順位を付けやすくなる
- 「べき論」に振り回されない
- チームで納得感のある判断ができる
::
