## エンジニアの役割の変化

### 「実装者」から「意思決定者」へ

収束状態において、エンジニアはコードを書く人から、判断を下す人へと移行する。ワークフローは以下のようになる。

::step-flow
::step{number="1" title="AI が提案" icon="mdi:robot"}
AI が複数の方針・設計案を提案
::

::step{number="2" title="人間が判断" icon="mdi:account-check"}
人間が採用・不採用を判断
::

::step{number="3" title="AI が実装" icon="mdi:code-braces"}
AI が自律的に実装・テスト・デプロイまで完了
::
::

::alert{type="info"}
従来の「発注者とベンダー」に似ているが、フィードバックが数秒〜数分で返るため、要求を対話的に詰めていくスタイルが有効になる。
::

## 人間と AI の役割分担

### 機能要件 → 人間がレビュー

「何をすべきか」はドメイン知識と意思決定の問題。人間が振る舞いの定義（より読みやすい形式のテスト）をレビューする。

### 非機能要件 → AI が自律的に最適化

パフォーマンス、セキュリティ、可用性は明確なメトリクスで定義できるため、AI が計測 → 評価 → 改善のループを自律的に回せる。

| 領域               | 人間                 | AI                         |
| ------------------ | -------------------- | -------------------------- |
| What（何を作るか） | 定義・レビュー・承認 | 提案・実装                 |
| How well（品質）   | 閾値の設定           | 自律的な検証・改善         |
| Why（判断理由）    | 意思決定・説明責任   | 選択肢とトレードオフの提示 |

## 現状と収束状態のギャップ

### 能力はある、枠組みがない

現在の生成 AI は個別タスクを高精度でこなせる。不足しているのは以下の枠組みである。

::info-card{title="不足している3つの枠組み" icon="mdi:puzzle"}

### 🔄 オーケストレーション層

複数タスクの実行順序制御、結果の受け渡し、失敗時のロールバック

### 💾 状態管理と継続性

長期プロジェクトの文脈を保持し、セッションをまたいで作業を継続する仕組み

### 🛡️ 信頼境界の定義

「ここまでは自律実行、ここからは人間に確認」という境界の宣言的な定義方法
::

::alert{type="success"}
枠組みが整った瞬間、変化は急速に進む可能性がある。iPhone が既存技術を一気に普及させたように、決定版となるツールが登場すれば業界標準になる。
::

## 収束状態の多様性

収束状態は一つではなく、プロダクト特性によって異なる姿になる。

::before-after{beforeTitle="高信頼性領域" afterTitle="迅速な試行錯誤領域"}
#before
**医療・防災・金融など**

人間によるレビューと承認が多く残り、デプロイ前確認が必須。

#after
**コンシューマアプリ・プロトタイプなど**

AI への委譲度が高く、自律デプロイも許容される。
::

## 残り続けるもの

::key-points{title="AIに置き換わらない人間の価値"}

- 課題の本質を見極める力
- 外部システムとの統合判断
- ドメイン知識の妥当性検証
- トレードオフの意思決定
- 判断の説明責任

::

## 結論

::alert{type="info"}
収束状態において、エンジニアの本質的な価値は **「What（何を作るか）」** と **「Why（なぜそう判断したか）」** に集約される。

この変化は技術的にはすでに実現可能な段階にあり、枠組みが整うのを待っている状態かもしれない。今から AI ファーストなワークフローを探求し、「判断者」としてのスキルを磨くことが、その波に乗るための準備となる。
::
